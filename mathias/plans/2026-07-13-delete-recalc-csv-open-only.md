# Fix: delete#recalc.csv must include only OPEN meldungen

## Context

The recalc bundle writes three CSVs per Grossfile: `*_return#recalc.csv`, `*_confirm#recalc.csv`,
and `*_delete#recalc.csv`. The confirm and delete files are meant to carry only successfully
processed ("OPEN") meldungen — no ERROR, no declined.

- **confirm** is correct: filtered by `StmStatus.isConfirmable()` → `OPEN` only.
- **delete** is wrong: filtered by `StmStatus.isDeletable()` → `OPEN || UPDATE_DECLINED || CONFIRM_DECLINED`,
  so two *declined* statuses leak into the delete file.

Legacy confirms this is a bug. In `~/dev/projects/oekb/ifas`, `c_st_meldung.cpp` writes **both** the
confirm and delete records under one identical gate (`ProcessMeldung_NEW` ~L2916, `ProcessMeldung_UPDATE` ~L3236):

```cpp
if (strcmp(pcStatusM->GetNewStatus(), "OPEN") == 0) {
    if (pfOutConfirm != NULL) WriteConfirmMeldung(...);
    if (pfOutDelete  != NULL) WriteDeleteMeldung(...);
}
```

Legacy only ever emitted OPEN meldungen to the delete file; declined ones were never written. The new
system diverged. Fixing it restores historical fidelity and keeps the recalc'd delete file from showing
spurious diffs against the legacy-produced original.

## Where the filter lives

- `RecalculationOutputs.writeDeleteCsv` (L360) → predicate `isDeletable` (L374) → `stm.getStatus().isDeletable() && stmId != null`
- `CalculationOutputs.writeDeleteCsv` (L112) → predicate `isDeletable` (L126) → `stm.getStatus().isDeletable() && referencedStmId != null`
- Both delegate to `StmStatus.isDeletable()` — the single shared source of the wrong status set.

`isDeletable()` has no other callers (verified by grep), so changing it is safe and fixes calc + recalc
together — matching legacy, which gated both paths on OPEN identically.

## Change

**File:** `ifas-database/ifas-persistence-stm/src/main/java/at/oekb/ifas/persistence/stm/steuermeldung/StmStatus.java`

Restrict `isDeletable()` to `OPEN` and update its Javadoc:

```java
/**
 * Whether an STM with this status may be emitted to the delete file.
 * Legacy (c_st_meldung.cpp ProcessMeldung_NEW/UPDATE) gated the delete file on new status == OPEN,
 * identically to the confirm file. Declined statuses were never written.
 * In addition to this status, the STM must also have an STM-ID.
 */
public boolean isDeletable() {
    return this == OPEN;
}
```

`isConfirmable()` already returns `OPEN`; leaving the two methods separate keeps the confirm-vs-delete
intent explicit even though the status set is now identical (mirrors the two separate legacy write calls).
The `stmId != null` / `referencedStmId != null` guards in the predicates are orthogonal and stay.

## Verification

1. `mvn clean install -Pno-proxy -pl ifas-database/ifas-persistence-stm -am` — compile the changed module.
2. Run recalc integration tests that exercise delete output and declined meldungen:
   - `mvn test -Pno-proxy -Dtest=QuickRecalculationTest`
   - `mvn test -Pno-proxy -Dtest=JiraIssueRecalculationTest` (has a CONFIRM_DECLINED scenario, L74)
   - `mvn test -Pno-proxy -Dtest=SteuerMeldungBundlesTest`
3. Confirm a recalc run's `*_delete#recalc.csv` no longer contains UDL/CDL meldungen — i.e. its meldung
   set now matches `*_confirm#recalc.csv`.

## Notes / risk

- No existing test pins the declined-inclusive behavior (the `DELETE_CSV_FILE` assertion in
  `SteuerMeldungBundlesTest` is about the *input* delete file and only checks presence/length).
- If any recalc golden-file fixture was generated by the *new* (buggy) system rather than legacy, its
  delete file may currently contain UDL/CDL rows and would need regenerating. Check test output; the
  legacy-sourced Grossfiles should already be OPEN-only and will now match.
